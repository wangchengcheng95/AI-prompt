# 系统总览

## 文档用途

本文档记录系统长期稳定的事实，用于帮助新会话快速理解服务的目标、边界、核心能力与外部关系。
默认适用于单仓库单服务的 Go 后端项目，也可作为仓库 Agent 的补齐指令。

## 补齐指引

- 优先依据已验证事实补齐，不根据命名习惯或个人经验臆测系统职责。
- 优先总结“服务是什么、负责什么、不负责什么”，避免落入代码实现细节。
- 如果信息不足，保留章节并显式写明 `待确认`，不要擅自补全。
- 如果发现与其他文档或代码存在冲突，先标记冲突点，再列出待核实来源。

## 优先信息来源

- 项目 `README`、产品说明或仓库首页描述
- 服务入口、路由注册、协议定义与对外暴露能力
- 与外部系统交互的客户端、SDK、消息主题或任务入口
- 已有设计文档中关于服务职责、边界和约束的稳定描述

## 完成标准

- 读者能在几分钟内理解该服务的目标、边界和核心能力。
- 读者能分清本服务负责什么、不负责什么，以及主要与谁交互。
- 文档内容以稳定事实为主，没有混入短期状态、运行步骤或代码细节。

## 缺失信息写法

- 未确认的信息写 `待确认`。
- 明确不存在的项写 `不适用`。
- 存在来源冲突的项写 `存在冲突，待核实`，并补充冲突来源。

## 建议阅读时机

- 新会话刚接手项目时优先阅读。
- 在判断某项需求、缺陷或改动是否属于本服务职责范围前阅读。

## 适合写入

- 系统目标与边界
- 不负责的范围
- 核心能力
- 主要外部交互方
- 长期稳定的约束条件

## 不适合写入

- 具体代码分层、包结构与调用链
- 启动命令、配置项清单、部署步骤与观测入口
- 当前迭代状态、短期待办或临时阻塞
