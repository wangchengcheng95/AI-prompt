# 设计决策记录

## 文档用途

本文档记录已经确认的重要设计决策及其原因，便于后续会话追溯取舍背景。
默认适用于单仓库单服务的 Go 后端项目，也可作为仓库 Agent 的补齐指令。

## 补齐指引

- 只写已经确认并实际生效的决策，不把开放讨论或临时想法写成结论。
- 优先解释“为什么这样做”，再写“具体影响了什么”。
- 如果只能确认结论、无法确认原因，应显式标注 `待确认`，不要擅自补因果链。
- 若同一决策在多个时期被更新，应新增条目说明新旧关系，不直接抹掉旧背景。

## 优先信息来源

- 设计文档、评审记录、PR 描述与提交信息
- 代码中的接口选择、目录调整、关键依赖引入或替换记录
- 团队约定、架构会议结论与现有 ADR
- 与重大改动相关的 issue、任务卡或讨论摘要

## 完成标准

- 读者能理解关键设计为何如此，而不仅知道当前实现长什么样。
- 每条记录都包含结论、原因、影响和可追溯入口。
- 文档没有混入当前进度跟踪、风险猜测或未定方案。

## 缺失信息写法

- 决策原因或影响无法确认时写 `待确认`。
- 项目尚未形成正式决策记录时写 `不适用`。
- 如果发现互相冲突的决策结论，写 `存在冲突，待核实`，并列出相关记录。

## 建议阅读时机

- 在准备调整既有设计、重构关键模块或评估替代方案前阅读。
- 在发现当前实现看似“不自然”但怀疑其背后有历史取舍时阅读。

## 适合写入

- 已确认的重要设计决策
- 选择某方案而非其他方案的原因
- 决策带来的影响范围
- 相关代码或文档入口

## 不适合写入

- 尚未收敛的开放讨论
- 当前迭代进展、阻塞或短期执行计划
- 未经验证的风险猜测或一次性实现细节

## 维护规则

- 仅记录已经确认的决策。
- 已否决方案不作为结论写入；如需保留背景，仅简要说明未采用原因。
- 如果条目逐渐增多或需要更严格追溯，可演进为 `adr/` 目录并采用“一条决策一份文件”。
- 如果内容主要描述当前推进情况，应写入 `current-status.md`。
- 如果与既有决策冲突，新增待确认冲突条目，不直接覆盖旧结论。
