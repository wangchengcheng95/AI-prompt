# 当前状态

## 文档用途

本文档记录当前阶段仍然有效的进展、阻塞与下一步，供新会话快速接续工作。
默认适用于单仓库单服务的 Go 后端项目，也可作为仓库 Agent 的补齐指令。

## 补齐指引

- 只保留当前仍然有效的状态，不把历史流水账原样堆积进来。
- 优先提炼“现在做到哪一步、卡在哪里、下一步接什么”。
- 若最近状态无法从仓库内直接判断，应明确说明观察依据不足，不要伪造进展。
- 如某条状态已经沉淀为长期事实或正式决策，应迁移到对应文档，不长期停留在这里。

## 优先信息来源

- 最近的提交记录、变更文件、未完成分支或 TODO
- issue、任务说明、设计草稿与当前会话上下文
- 最新代码状态、测试结果、临时脚本和待处理注释
- 其他文档里提到但尚未完成的事项

## 完成标准

- 读者能快速判断当前阶段的完成情况、阻塞点和下一步。
- 条目内容是“当前有效状态”，而不是模糊背景介绍或历史复读。
- 读者能据此接续工作，而不必先重新梳理最近一轮上下文。

## 缺失信息写法

- 无法确认的进展或阻塞写 `待确认`，并尽量说明原因。
- 当前确实没有该类信息时写 `不适用`。
- 如果不同来源对当前状态描述不一致，写 `存在冲突，待核实`。

## 建议阅读时机

- 在准备承接当前迭代、继续未完成工作或判断最近优先级时阅读。
- 在确认哪些信息仍然有效、哪些事项正在推进时阅读。

## 适合写入

- 当前阶段已完成的重要进展
- 正在进行中的工作
- 已确认的阻塞点
- 下一步建议动作

## 不适合写入

- 长期稳定的系统事实或架构边界
- 已经定稿的设计决策结论
- 未经确认的风险猜测或长期沉淀的风险事实

## 记录格式

### YYYY-MM-DD 状态标题

- 已完成：
- 进行中：
- 阻塞点：
- 下一步：

## 维护规则

- 只记录当前仍然有效的阶段状态。
- 已失效状态应通过更新条目替换，不保留重复叙述。
- 如果内容已经成为长期稳定事实，应迁移到 `system-overview.md`、`architecture.md` 或 `operations.md`。
- 如果内容是已经确认的取舍结论，应迁移到 `decisions.md`。
- 如果工作区存在无关脏改动，仅记录本次任务范围内已确认的状态。
