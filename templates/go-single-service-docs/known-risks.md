# 已知风险

## 文档用途

本文档记录已经确认的风险、脆弱点、限制条件与当前规避方式，用于排障、重构与高风险变更前的快速检查。
默认适用于单仓库单服务的 Go 后端项目，也可作为仓库 Agent 的补齐指令。

## 补齐指引

- 只写已经验证、重复出现或被明确观察到的风险，不写模糊担忧。
- 优先描述风险触发条件、影响范围和当前规避方式，避免只写结论不写上下文。
- 如果风险已经解除，补充关闭说明；如果仍存在但缺少细节，明确写 `待确认`。
- 不要把“当前没做完的事情”直接当风险，除非已经确认其会带来明确影响。

## 优先信息来源

- 线上事故复盘、bug 记录、历史缺陷与排障文档
- 测试中暴露的脆弱点、已知限制和兼容性说明
- 监控告警、超时重试、幂等、并发、数据一致性相关说明
- 代码中的 FIXME、保护性注释和已知限制描述

## 完成标准

- 读者能看懂主要风险是什么、何时触发、影响哪里、如何规避。
- 风险条目足够具体，能支撑改动前检查和排障判断。
- 文档没有混入纯猜测、短期状态或完整设计讨论。

## 缺失信息写法

- 风险细节未确认时写 `待确认`。
- 当前确实没有已验证风险时写 `不适用`。
- 如果不同记录对风险描述不一致，写 `存在冲突，待核实`，并补充来源。

## 建议阅读时机

- 在修改关键链路、处理线上问题或进行高风险重构前阅读。
- 在评估某项改动是否可能触发历史坑点或系统脆弱点时阅读。

## 适合写入

- 已验证的风险与脆弱点
- 风险影响范围
- 风险触发条件
- 当前规避方式与建议后续处理

## 不适合写入

- 一次性异常、临时猜测或未复现的问题
- 当前迭代进展、待办事项或执行计划
- 设计方案的完整背景与取舍过程

## 维护规则

- 只记录已经验证或被重复观察到的风险。
- 临时猜测、一次性异常、未确认问题不写入本文档。
- 如果风险已经解除，应补充关闭说明，而不是直接删除背景。
- 如果内容主要是当前阶段阻塞，应写入 `current-status.md`；如果内容主要是设计取舍，应写入 `decisions.md`。
