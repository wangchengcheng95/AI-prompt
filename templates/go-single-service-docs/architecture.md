# 架构说明

## 文档用途

本文档记录系统内部结构、分层边界、关键关系与主要扩展点，用于支持跨模块改动、设计评审与后续重构。
默认适用于单仓库单服务的 Go 后端项目，也可作为仓库 Agent 的补齐指令。

## 补齐指引

- 优先从真实代码结构和调用关系归纳架构，不根据目录命名直接推断职责。
- 先写稳定的模块边界和关键链路，再写扩展点或可替换点。
- 对仍在演进中的结构，只记录当前已确认部分；未定方案不要当成既定事实。
- 如果模块职责跨界或边界不清，先标 `待确认`，避免输出看似完整但不准确的分层描述。

## 优先信息来源

- `cmd/`、`main.go`、服务入口与初始化流程
- 目录结构、核心 package、interface 定义与依赖注入位置
- handler / service / repository / job / consumer 等关键调用链
- 架构图、设计说明、测试用例中体现的模块职责与协作方式

## 完成标准

- 读者能快速看懂系统的主要模块、职责边界和关键调用链。
- 读者能判断某类改动应该落在哪一层、会影响哪些模块。
- 文档没有混入运行步骤、部署说明或短期迭代状态。

## 缺失信息写法

- 未确认的模块职责或调用关系写 `待确认`。
- 项目中确实不存在的层或扩展点写 `不适用`。
- 多处实现不一致时写 `存在冲突，待核实`，并列出相关代码入口。

## 建议阅读时机

- 在开始阅读代码或准备修改跨模块逻辑前阅读。
- 在判断某个能力落在哪一层、某段调用链应如何调整前阅读。

## 适合写入

- 总体结构
- 核心模块与职责
- 关键调用链或数据流
- 依赖关系与边界
- 扩展点与可替换点

## 不适合写入

- 服务职责边界、业务范围界定与非目标能力
- 启动方式、环境变量、部署流程与排障入口
- 当前迭代状态、短期任务安排或未确认的方案讨论

- 架构调整时同时写清影响范围与保持不变的约束。
